# AxiMate 开源组件使用规划与 Apache-2.0 遵从约束

## 1. 文档目的与适用范围

**目的**：规范 AxiMate 对基于阿里云原生生态的**上游开源组件**的引入、开发、构建与分发行为，确保对 **Apache License 2.0**（Apache-2.0）的**可追溯遵从**，降低合规与品牌风险。

**适用范围**：

- **Gateway 层**（基于 Higress）
- **Orchestrator 层**（基于 HiClaw）
- **Worker 层**（基于 CoPaw / ClawWorker 等 MCP 技能运行时）

以及上述组件所**随附或捆绑分发**的依赖（含传递依赖中以 Apache-2.0 或需一并声明的其他许可证授权的软件）。

**前提**：各上游主组件以各仓库 `LICENSE` 及发布物为准；若实际许可证与 Apache-2.0 不一致，以**实际许可证**为准并更新本文档与 SBOM。

---

## 2. Apache-2.0 核心义务（工程必须落实）

在**再分发**上游软件（源码或二进制，含容器镜像层、安装包、交付给客户的制品）时，通常需满足包括但不限于：

1. **保留版权声明与许可全文**  
   随分发物提供 Apache-2.0 全文；保留上游版权行及许可声明。

2. **NOTICE 文件**  
   若上游提供 `NOTICE`，再分发时须保留；AxiMate 自有 `NOTICE` 应追加而非覆盖上游要求的内容（合并方式由法务/发布规范定稿）。

3. **修改声明（State Changes）**  
   对上游源码的**实质性修改**若再分发，应合理标注变更（`NOTICE`、变更日志、发行说明等，措辞由法务审定）。

4. **专利与商标**  
   尊重许可证中的专利授权边界；不暗示未获授权的上游背书，遵守各社区商标政策。

5. **场景区分**  
   自托管使用与向客户交付制品所触发的义务不同；须在发布清单中区分并由法务确认具体场景。

## 3. 分层规划与合规落点

| 层级 | 规划要点 | Apache-2.0 遵从落点 |
|------|----------|---------------------|
| Gateway | 版本锁定、可重现构建、Wasm 边界清晰 | 分发网关/Wasm 时附带 LICENSE/NOTICE；Fork 修改记录变更 |
| Orchestrator | 依赖清单、API 稳定性 | 发布镜像/包时合并 NOTICE；文档声明上游版本 |
| Worker | MCP 技能与运行时依赖分离 | 技能包含上游代码时随包或发布页提供许可证与归属 |

## 4. 工程约束

### 4.1 SBOM 与许可证台账

- 维护 SBOM（推荐 SPDX 或 CycloneDX）：组件名、版本、来源 URL、许可证、是否修改版、是否存在 `NOTICE`。
- 禁止许可证不明或与策略冲突的依赖进入发布产物；CI 对发布产物做许可证扫描（策略由安全/法务制定）。

### 4.2 源码与 Fork

- Fork 上游须标明来源与基线 tag/commit；发布从可追溯标签构建。
- Vendor 或内嵌源码须保留上游许可证头与 `NOTICE`。

### 4.3 构建与发布

- **容器镜像**：镜像内或发布说明提供 `LICENSE`、`NOTICE`；列出主要 Apache-2.0 组件及版本。
- **二进制 / Wasm**：随发布包或固定 URL 提供许可证文本；修改上游模板须记录变更说明。
- **前端等非 Apache-2.0 依赖**（如 MIT）：在 SBOM 中**分别标注**，并按各自许可证保留声明。

### 4.4 对外文档

- README/白皮书注明 “Powered by / includes” 的上游名称与版本；提供第三方开源声明页面或章节。
